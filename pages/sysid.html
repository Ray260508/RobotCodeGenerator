Tuning with WPILib SysId

Instead of guessing PID and Feedforward values, the generated code includes a SysIdRoutine for your configured mechanisms. This allows you to characterize real robot behavior and compute reliable kS, kV, kA and starter PID gains.

Step 1 Prepare Hardware and Safety

  • WPILib baseline: Test one mechanism at a time, with enough free travel and clear field space.
  • CTRE setup: Verify neutral mode, current limit, and inversion in Phoenix Tuner X before running tests.
  • REV setup: Confirm idle mode, current limits, and sensor phase in REV Hardware Client first.
  • 6328-style logging: Ensure logging is stable (AdvantageKit/WPILib logs) before tuning so bad logs do not produce bad constants.

Step 2 Bind and Run SysId Commands

In your generated RobotContainer.java, bind the mechanism SysId commands to safe driver controls.

// Example: Elevator SysId bindings
controller.povUp().whileTrue(elevator.sysIdQuasistatic(SysIdRoutine.Direction.kForward));
controller.povDown().whileTrue(elevator.sysIdQuasistatic(SysIdRoutine.Direction.kReverse));
controller.povRight().whileTrue(elevator.sysIdDynamic(SysIdRoutine.Direction.kForward));
controller.povLeft().whileTrue(elevator.sysIdDynamic(SysIdRoutine.Direction.kReverse));

Run all four tests if possible:

  • Quasistatic Fwd/Rev: voltage ramp for kS + kV.
  • Dynamic Fwd/Rev: step input for kA.

Step 3 Analyze and Handle Common Situations

Load the .wpilog into SysId from VS Code tools and inspect outliers before accepting gains.

  1. Arm/elevator gravity effects: if forward/reverse differ a lot, keep distinct expectations and verify gearing/friction.
  2. Shooter/flywheel: prioritize clean velocity traces; repeat tests if note slips or balls contact the wheel.
  3. Drive characterization: keep wheels on consistent carpet/tile during one data set; mixed surfaces skew results.
  4. Noisy data: reduce mechanical slop, check wiring, and rerun instead of forcing aggressive gains.

Step 4 Apply Constants and Validate On-Robot

Copy results into generated Constants.java, then validate with small command tests before full-speed operation.

  • Start with lower output limits and increase gradually.
  • Watch current draw and thermal behavior.
  • Retune after major hardware changes (motor swap, gear ratio change, mechanism rebuild).
